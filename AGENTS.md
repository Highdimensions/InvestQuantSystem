# Repository Guidance

## Project purpose

本仓库用于开发短线量化交易建议与有效性评价系统。

当前系统只产生研究性 Buy、Sell、Hold 建议并进行影子评价，
不连接真实券商账户，不自动执行交易。

## Working rules

- 修改前先阅读与任务相关的已评审设计文档，优先入口为 `docs/architecture/development-guide.md` 和 `PLANS.md`。
- `docs/input/original-quant-signal-plan.md` 只作为历史输入留档；仅在需要追溯原始需求或核对设计来源时阅读。
- 对复杂任务先输出执行计划，再修改文件。
- 不得将回测结果描述为收益承诺。
- 明确区分信号研究、模拟成交和真实交易。
- 所有价格评价必须基于信号产生后现实可获得的价格。
- 不允许使用未来数据、未来修订数据或信号时刻尚未闭合的 K 线。
- 不修改历史 SignalEvent，只能通过版本创建新信号。
- 实时和回测应复用相同的特征及策略核心。
- 新增模块时说明职责、输入、输出、依赖和失败处理。
- 所有架构结论必须标明“已确定”“建议方案”或“待决策”。

## Documentation rules

- 文档正文使用中文。
- 类名、接口名、事件名和字段名使用英文。
- 使用 Mermaid 表达系统结构、时序和状态机。
- 不为了显得完整而虚构业务需求、性能数据或团队制度。
- 信息不足时记录到“开放问题”，不要自行假设为事实。

## Definition of done

架构、计划或工程修改完成时，应当：

1. 给出清晰的系统边界和非目标。
2. 完成模块职责及依赖划分。
3. 定义关键数据契约。
4. 说明实时、回测和评价链路。
5. 给出测试、可观测性和演进方案。
6. 列出风险、待决策事项和实施阶段。
7. 对复杂或高风险变更完成一次架构、量化正确性及可测试性复核。
